# 任务聚焦最佳实践

## 核心原则

### 原则 1：早期创建

> **最重要的原则：任务一开始就创建 todo.md**

| 时机 | 效果 |
|-----|------|
| 任务开始立即创建 | ✅ 目标清晰、方向明确 |
| 任务中途创建 | ❌ 可能已偏离目标 |

### 原则 2：保持简洁

todo.md 是**导航工具**，不是操作日志。

| 避免 | 推荐 |
|-----|------|
| `- [ ] 编写 softmax 函数，使用 ReduceMax 找最大值，然后用 Exp 计算指数...` | `- [ ] 编写 softmax 核心逻辑` |
| `- [ ] 和用户讨论了需求，用户说要支持 FP16，所以修改代码...` | `- [ ] 添加 FP16 支持` |

### 原则 3：持续更新

| 频率 | 说明 |
|-----|------|
| 每完成一步 | 立即勾选并更新进度 |
| 每 3-5 步 | 重新读取并打印 todo.md |
| 任务开始/结束 | 明确标记开始和完成 |

### 原则 4：保持在末尾

> **核心机制：将 todo.md 内容反复写入上下文末尾**

这确保目标始终在模型的近期注意力范围。

## 使用场景决策树

```
收到任务
    │
    ├─ 步骤 < 3？
    │   └─ 是 → 不需要 todo.md
    │
    ├─ 快速问答？
    │   └─ 是 → 不需要 todo.md
    │
    └─ 复杂任务 (>3 步)
        └─ 是 → 立即创建 todo.md
```

## 步骤粒度指南

### 合适的粒度

每个步骤应该在 **10-30 分钟**内可完成。

| 粒度过细 | 适中 | 粒度过粗 |
|---------|------|---------|
| 打开文件 | 编写 kernel 代码 | 实现算子 |
| 输入代码 | 编译测试 | 完成开发 |
| 保存文件 | 验证精度 | 修复问题 |

### 步骤命名模式

使用 **动作 + 对象 + 说明** 格式：

```markdown
- [ ] [动作] [对象]：[说明]
```

示例：
```markdown
- [ ] 编写 kernel 代码：实现 sinh 计算逻辑
- [ ] 编译测试：验证代码无语法错误
- [ ] 精度验证：对比 FP16/FP32 结果
```

## 更新节奏

### 标准节奏

```
完成步骤 → 更新 todo.md → 继续下一步
              ↓
         每 3-5 步 → 打印 todo.md 到上下文末尾
```

### 打印时机

| 情况 | 是否打印 |
|-----|---------|
| 完成单个步骤 | ❌ |
| 每 3-5 步 | ✅ |
| 切换阶段 | ✅ |
| 发现阻塞 | ✅ |
| 任务过半 | ✅ |
| 任务完成 | ✅ |

## 常见错误

### 错误 1：创建过晚

**症状**：任务进行到一半才发现偏离目标

**解决**：任务开始立即创建 todo.md

### 错误 2：过度详细

**症状**：todo.md 变成了操作日志

**解决**：保持步骤粒度适中，每步 10-30 分钟

### 错误 3：长期不更新

**症状**：todo.md 创建后就没再看过

**解决**：每 3-5 步就重新打印

### 错误 4：不放在末尾

**症状**：创建了 todo.md 但注意力还是分散

**解决**：每次更新都将内容写入上下文末尾

## 高级技巧

### 技巧 1：渐进式分解

适合：开始时不确定所有步骤

```markdown
## 待办事项
### 已明确
- [ ] 需求分析

### 待明确（需求分析后补充）
- [ ] API 调研
- [ ] 方案设计
```

### 技巧 2：阶段式组织

适合：有明显阶段划分的任务

```markdown
### 阶段 1：需求分析 (2/2) ✅
### 阶段 2：API 调研 (1/4)
### 阶段 3：方案设计 (0/3)
```

### 技巧 3：调试计数

适合：精度调试等需要计数的场景

```markdown
## 调试计数
3/7

如果计数 >= 7，切换到二分调试
```

### 技巧 4：阻塞处理

遇到阻塞时明确记录：

```markdown
## 阻塞问题
- [ ] API 文档不清楚 → 等待查询
- [ ] 编译失败 → 需要查看错误日志
```

### 技巧 5：动态调整

```markdown
## 调整记录
- [原方案] 直接使用 FP16
- [新方案] FP32 中间计算
- 原因：API 不支持
```

## 质量检查清单

完成 todo.md 后，检查：

- [ ] 目标清晰明确
- [ ] 步骤数量适中 (5-20 个)
- [ ] 每个步骤 10-30 分钟可完成
- [ ] 使用动作动词开头
- [ ] 包含进度跟踪
- [ ] 预留阻塞问题区域

## 注意力管理要点

### 为什么要在末尾？

| 位置 | 注意力效果 |
|-----|-----------|
| 上下文开头 | ⚠️ 模型可能遗忘 |
| 上下文中间 | ❌ 迷失在中间 |
| **上下文末尾** | ✅ 近期注意力范围 |

### 打印格式

建议每次打印包含：

```markdown
=== 任务焦点 ===
当前任务：[任务名称]
当前阶段：[阶段名称]
进度：X/Y (Z%)
下一步：[下一个步骤]
================
```
