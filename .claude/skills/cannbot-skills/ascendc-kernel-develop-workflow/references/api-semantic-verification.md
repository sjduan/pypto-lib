# API 语义验证方法论（通用）

> **核心原则**：任何 API 调用前，必须理解数据布局和 API 能力，确保匹配

---

## 问题本质

**SubAgent 为什么无法发现 API 错误？**

❌ **只检查语法**：参数个数、类型、顺序
✅ **应该检查语义**：API 能否实现预期功能？数据布局与 API 能力是否匹配？

**典型案例**：

| 错误类型 | 错误原因 | 后果 |
|---------|---------|------|
| API 能力不匹配 | 数据带 stride，但用了只能处理连续数据的 API | 编译通过，运行时错误 |
| 违反限制条件 | 数据未对齐，但用了要求对齐的 API | 运行时崩溃或结果错误 |
| 性能低效 | 用了低效 API，有更好的选择 | 性能差，浪费资源 |

---

## 通用验证方法论（3 问法）

**任何 API 调用前，必须回答以下 3 个问题**：

```
问题 1：数据布局是什么？
    ├─ 内存如何排列？（连续 / 带 stride / 矩阵 / 多维）
    ├─ 是否对齐？（32 字节对齐 / 无对齐 / 其他）
    └─ 输入输出格式？（标量 / 向量 / 矩阵）

问题 2：需要什么操作？
    ├─ 操作类型？（reduce / broadcast / elementwise / copy / cast / ...）
    ├─ 操作维度？（沿哪个维度、多少元素、是否需要 stride）
    └─ 特殊要求？（数值稳定 / 高精度 / 特定范围）

问题 3：API 能实现吗？
    ├─ 查阅官方文档了吗？（asc-devkit/docs/api/context/）
    ├─ API 适用场景对吗？（根据文档说明判断）
    ├─ 满足 API 限制吗？（对齐、类型、取值范围等）
    └─ 有更好的选择吗？（性能、易用性）
```

**验证公式**：

```
正确使用 API = (数据布局 ∈ API 支持范围) AND (满足所有限制条件) AND (无更好选择)
```

## 强制验证步骤（适用于所有 API）

### 步骤 1：查阅官方文档

**必须执行**：

```bash
# 查阅 API 文档
asc-devkit/docs/api/context/{API_NAME}.md

# 查阅示例代码（如果不确定）
asc-devkit/examples/**/*{api_name}*
```

**重点关注**：
- API 签名（参数类型、顺序、返回值）
- **适用场景**（API 能做什么，不能做什么）
- **限制条件**（对齐、类型、取值范围等）
- **使用示例**

### 步骤 2：确认数据布局

**必须明确**：

```markdown
输入数据：
- 内存布局：{连续 / 带 stride / 矩阵 / 多维}
- 对齐状态：{32 字节对齐 / 未对齐}
- 维度信息：{Shape、每个维度的含义}

输出数据：
- 内存布局：{连续 / 带 stride / ...}
- 维度信息：{Shape}
```

### 步骤 3：验证匹配性

**必须验证**：

```markdown
匹配验证表：

| 验证项 | 数据实际情况 | API 要求 | 是否匹配 |
|--------|------------|---------|---------|
| 数据连续性 | {连续 / stride} | {连续 only / 支持 stride} | {是/否} |
| 对齐要求 | {已对齐 / 未对齐} | {必须对齐 / 无要求} | {是/否} |
| 数据类型 | {FP32/FP16/...} | {支持类型} | {是/否} |
| 取值范围 | {范围} | {限制范围} | {是/否} |
```

**如果有任何一项不匹配 → 必须换 API 或调整数据**

### 步骤 4：确认选择

**必须确认**：

- [ ] 已查阅官方 API 文档
- [ ] 数据布局与 API 能力匹配
- [ ] 满足 API 的所有限制条件
- [ ] 无更好的 API 选择（或已说明选择原因）

### 步骤 5：记录验证结果

**必须记录**：

```markdown
API 验证记录：

**API 名称**: {API_NAME}
**官方文档**: [链接]
**数据布局**: {描述}
**功能需求**: {描述}
**验证结果**: ✅ 通过 / ❌ 不通过
**选择原因**: {说明}
```

## 常见错误模式

### 错误 1：未查阅文档就使用 API

```
❌ 凭记忆写 API 参数
❌ 不确定 API 是否支持某个功能就使用
❌ 未确认 API 限制条件

✅ 必须先查阅官方文档
✅ 确认 API 适用场景和限制
✅ 参考官方示例代码
```

### 错误 2：数据布局与 API 能力不匹配

```
❌ 数据带 stride，但用了只能处理连续数据的 API
❌ 数据未对齐，但用了要求对齐的 API
❌ 需要广播，但选了不支持广播的 API

✅ 必须确认数据布局
✅ 必须验证 API 能力范围
✅ 必须确保匹配
```

### 错误 3：未考虑更好选择

```
❌ 有更高效的 API，但用了低效的
❌ 有更简单的 API，但用了复杂的
❌ 有专门的 API，但用了通用的

✅ 查阅是否有更好的 API
✅ 对比性能和易用性
✅ 选择最优方案
```

## 验证决策流程（通用）

```
开始：需要使用某个 API
    ↓
【步骤 1】查阅官方文档
    ├─ 确认 API 签名
    ├─ 确认适用场景
    └─ 确认限制条件
    ↓
【步骤 2】确认数据布局
    ├─ 内存排列方式
    ├─ 对齐状态
    └─ 维度信息
    ↓
【步骤 3】验证匹配性
    ├─ 数据布局匹配？
    │   ├─ 否 → ❌ 换 API 或调整数据
    │   └─ 是 ↓
    ├─ 限制条件满足？
    │   ├─ 否 → ❌ 换 API 或调整数据
    │   └─ 是 ↓
    └─ 有更好选择？
        ├─ 是 → 考虑换 API
        └─ 否 ↓
【步骤 4】确认选择
    ↓
【步骤 5】记录验证结果
    ↓
✅ 使用 API
```

## 设计时如何应用

在设计文档中，对于每个关键 API，必须填写：

```markdown
### API 验证表

**API 名称**: {API_NAME}
**官方文档**: [链接](asc-devkit/docs/api/context/{API_NAME}.md)

#### 1. 数据布局
- **输入**: {内存布局、对齐状态、维度}
- **输出**: {内存布局、维度}

#### 2. 功能需求
- **操作类型**: {reduce/broadcast/elementwise/copy/...}
- **操作维度**: {描述}
- **特殊要求**: {如有}

#### 3. API 能力（根据文档）
- **适用场景**: {从文档复制}
- **限制条件**: {从文档复制}

#### 4. 匹配验证
- [ ] 数据布局匹配
- [ ] 限制条件满足
- [ ] 无更好选择（或已说明）

#### 5. 验证结论
- ✅ 通过 / ❌ 不通过
- **选择原因**: {说明}
```

## 代码审查时如何验证

在代码审查时，对于每个 API 调用，SubAgent 必须执行：

```markdown
## API 语义验证

### {API_NAME}

**1. 是否查阅官方文档？**
   - [ ] 是（提供链接）
   - [ ] 否 → ⛔️ 必须查阅

**2. 数据布局是否确认？**
   - [ ] 是（描述布局）
   - [ ] 否 → ⛔️ 必须确认

**3. 匹配性是否验证？**
   - [ ] 数据布局匹配
   - [ ] 限制条件满足
   - [ ] 无更好选择

**4. 验证结论**
   - ✅ 通过
   - ❌ 不通过（说明原因）

**如果任何一项未通过，代码审查失败** ⛔️
```

## 核心思维模型

**最关键的思维**：

```
不要问：这个 API 怎么用？
而要问：这个 API 能做什么？我的数据需要什么？两者匹配吗？
```

**验证公式**：

```
正确使用 = 理解数据 + 理解 API + 确认匹配
```

**一句话总结**：

**任何 API 调用前，先问 3 个问题：数据布局是什么？需要什么操作？API 能实现吗？**
