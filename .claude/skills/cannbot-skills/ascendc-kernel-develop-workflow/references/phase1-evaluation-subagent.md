### 8.2 通用 Agent：评估设计文档

设计文档完成后，**由主 Agent 调度通用 Agent** 评估设计文档质量。

**评估 Prompt 模板**：
```
你是 Ascend C 算子设计文档评审专家。请评估 {operator_name} 的设计文档。

**设计文档**：`ops/{operator_name}/docs/design.md`

**评估维度（10分制）**：

| 维度 | 分值 | 说明 |
|-----|------|------|
| 设计清晰度 | 2分 | 数学公式、数据流、API 映射是否清晰 |
| **多核切分策略** | **2分** | **核数计算是否动态（禁止写死）、切分维度是否合理、负载均衡是否考虑** |
| **分支伪代码完整性** | **3分** | **每个分支（2.4节）是否有完整的核心伪代码** |
| 技术细节 | 1分 | 优化技巧、对齐处理、Buffer 规划是否明确 |
| 完整性 | 1分 | 是否覆盖所有必要设计要素 |
| 正确性 | 1分 | 逻辑是否正确，是否有明显错误 |

**⚠️ 多核切分策略重点检查项**：
- [ ] 设计文档中是否明确说明切分维度（按行/按列/按 tile）
- [ ] 核数计算是否使用动态计算（根据算子类型选择正确的API + 条件判断）
- [ ] **是否禁止写死核数**（❌ `numBlocks = 8;` → ✅ 动态计算）
- [ ] 是否考虑负载均衡（如：数据量小于核数时的处理）
- [ ] 是否在 Kernel 侧有越界检查（`if (blockIdx >= usedCoreNum) return;`）

**核数获取 API 选择**：
| 算子类型 | 使用的 API |
|---------|-----------|
| 纯向量计算 | `ACL_DEV_ATTR_VECTOR_CORE_NUM` |
| 矩阵计算 | `ACL_DEV_ATTR_CUBE_CORE_NUM` |
| 混合计算 | `ACL_DEV_ATTR_AICORE_CORE_NUM` |

**核数获取 API 选择**：
| 算子类型 | 使用的 API |
|---------|-----------|
| 纯向量计算 | `ACL_DEV_ATTR_VECTOR_CORE_NUM` |
| 矩阵计算 | `ACL_DEV_ATTR_CUBE_CORE_NUM` |
| 混合计算 | `ACL_DEV_ATTR_AICORE_CORE_NUM` |

**⚠️ 重点检查**：
1. **多核切分策略是否合理**（核数动态计算、切分维度、负载均衡）
2. 2.4节中是否每个分支都有独立的完整核心伪代码（不能是引用）
3. 核心伪代码只包含 Compute 核心计算流程（CopyIn/CopyOut 非必需）
4. 伪代码是否有 Buffer 使用标注
5. 伪代码是否有 API 选择说明
```

### 8.3 评分标准

| 分数 | 含义 | 说明 |
|------|------|------|
| **9.0-10** | 优秀 | 可以直接编码，无需补充 |
| **8.5-8.9** | 良好 | 可以编码，有少量不明确处但不影响主体实现 |
| **7.0-8.4** | 及格 | 有较多问题，需要补充后才能编码 |
| **< 7.0** | 不合格 | 存在严重问题，无法编码 |

### 8.3 质量控制循环

```
评估设计文档
    ↓
评分 >= 8.5？
    ├─ Yes → 设计完成，准出进入阶段二
    └─ No  → 根据反馈完善设计文档
              ↓
           重新评估（循环直到 >= 8.5）
```

**完善重点**：
- 🔴 **严重问题**：逻辑错误、分支伪代码缺失 → 必须修复
- 🟡 **中等问题**：参数说明不足、实现细节缺失 → 建议补充
- 🟢 **轻微问题**：示例缺失、文档格式 → 可选改进
