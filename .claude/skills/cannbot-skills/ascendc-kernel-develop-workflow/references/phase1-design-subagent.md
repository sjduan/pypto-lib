> **目标读者**：通用 Agent（由主 Agent 通过 Task 工具调度）
> 
> **用途**：执行算子设计步骤 1-7，输出设计文档（design.md）

---

## 1. 需求检查

### 必需信息

| 类别 | 检查项 | 必需 | 验证方法 |
|-----|--------|------|----------|
| **基本信息** | 算子名称、功能描述 | ✅ | 用户明确提供 |
| **数学定义** | 数学公式、计算规则、边界条件 | ✅ | 公式或伪代码 |
| **输入输出** | Shape、数据类型、广播规则 | ✅ | 明确列出 |
| **性能精度** | 精度要求、性能指标、数据规模 | ✅ | 提供误差容忍度 |
| **环境信息** | 服务器类型、CANN 版本 | ✅ | 明确说明 |

### 常见算子类型模板

| 算子类型 | 典型公式 | 输入数量 | 特殊要求 |
|---------|---------|---------|---------|
| 一元运算 | `y = f(x)` | 1 | 与输入相同 |
| 二元运算 | `z = f(x, y)` | 2 | 确认广播规则 |
| 归约运算 | `y = Σ(x[i])` | 1 | 指定归约维度 |
| 矩阵运算 | `C = A × B` | 2 | 通常需要 Tiling |

## 5. API 可行性验证

### 5.1 API 存在性验证

- 将公式拆解为 Ascend C API 的组合
- 查阅文档验证 API 存在性和数据类型支持
- 本地文档不足时调用 `/ascendc-docs-search`

### 5.2 关键 API 参数语义验证 ⚠️ **强制**

**适用范围**：所有带复杂参数的 API（Pattern、BinaryRepeatParams、DataCopyPad、Cast 等）

**强制验证步骤**：

#### 步骤 1：识别关键 API

**判断标准**：
- API 有多个可选参数（如 Pattern、BinaryRepeatParams）
- 参数值会影响计算语义（如沿哪一维 Reduce）
- 参数与数据布局相关（如 stride、Pattern）

#### 步骤 2：填写数据布局 → API 参数映射表

**必须填写**（在设计文档中）：

```markdown
## 数据布局 → API 参数映射

### Reduce API（如适用）
| 项目 | 值 | 说明 |
|-----|---|------|
| 输入数据布局 | (R, A0) 或 (A0, R) | 第一维是哪个？ |
| Reduce 维度 | 第一维 或 第二维 | 沿哪一维 Reduce？ |
| Pattern 选择 | RA 或 AR | 必须与 Reduce 维度匹配 |
| 官方文档 | [链接] | 已查阅 [ ] |

### 算术运算 API（如适用）
| 项目 | 值 | 说明 |
|-----|---|------|
| 操作类型 | 标量 或 广播 | 单行 or 多行？ |
| 广播参数 | src1RepStride=0 | 如需广播 |
| 分批大小 | 64 | R > 64 时分批 |
| 官方文档 | [链接] | 已查阅 [ ] |
```

#### 步骤 3：对照官方文档验证

**必须查阅**：`asc-devkit/docs/api/context/{API名称}.md`

**验证清单**：
- [ ] 已找到 API 的官方文档
- [ ] 已阅读"参数说明"章节
- [ ] 已理解关键参数的语义
- [ ] 已确认参数与数据布局的匹配关系
- [ ] 已在设计文档中记录

#### 步骤 4：参考官方示例

**查找方法**：
```bash
grep -r "API名称" asc-devkit/examples/ --include="*.asc" -A 3 | head -20
```

#### 步骤 5：记录验证结果

**在设计文档中必须包含**：
```markdown
## API 参数验证记录

**Reduce API 验证**：
- 官方文档：`asc-devkit/docs/api/context/{API}.md`
- 数据布局：(R, A0)
- Pattern 选择：RA
- 验证结论：✅ 匹配
```

**⚠️ 警示**：
- ❌ 禁止凭直觉或记忆选择 API 参数
- ❌ 禁止不查阅官方文档就确定参数值
- ✅ 必须通过"数据布局 → API 参数"映射验证

## 6. 分支规划与文件组织

### 6.1 判断是否需要分支

**简单算子（单文件）**：
- 无明显分支场景（对齐/非对齐、大shape/小shape等）
- 实现逻辑简单（< 300行代码）
- 示例：基础加减乘除、逐元素激活函数

**复杂算子（分支独立文件）**：
- 有多个分支场景（不同dtype、不同shape、不同对齐情况）
- 每个分支逻辑复杂（> 200行代码）
- **每个分支都要有独立的核心流程伪代码**
- 示例：Softmax（AR/ARA模板）、LayerNorm（对齐/非对齐）、Reduce（全载/分载）

### 6.2 文件目录规划

**简单算子**：
```
{operator_name}/
├── {operator_name}.asc           # 主入口 + 所有实现
├── CMakeLists.txt
├── gen_golden.py
└── run.sh
```

**复杂算子**：
```
{operator_name}/
├── {operator_name}.asc           # 主入口 + 分支判断框架
├── {operator_name}_common.h      # 公共定义（常量、工具函数）
├── {operator_name}_{分支1}.h     # 分支1（Tiling结构体 + 判断函数 + Kernel实现）
├── {operator_name}_{分支2}.h     # 分支2
├── CMakeLists.txt
├── gen_golden.py
└── run.sh
```

### 6.3 分支文件结构规范

每个分支 `.h` 文件必须包含三部分：

```cpp
// 1. Tiling 结构体（独立定义）
struct {operator_name}{分支}Tiling {
    uint32_t param1;
    uint32_t param2;
};

// 2. 判断函数（判断是否应走此分支）
__inline__ bool Is{operator_name}{分支}Case(参数列表) {
    return 判断条件;
}

// 3. Kernel 实现
class Kernel{operator_name}{分支} {
public:
    __aicore__ inline void Process(...);
};
```

### 6.4 Tiling 参数设计

**识别关键 Tiling 参数**：
- 多核切分：总任务如何分配给各核
- UB 切分：单核如何分 chunk 处理
- 对齐参数：32字节对齐后的长度

## 7. 输出设计文档

**必须输出设计文档**：`{operator_name}/docs/design.md`

**使用模板**：[design-template.md](../templates/design-template.md)

### 7.1 核心流程伪代码要求（复杂算子强制）

**⚠️ 重要：1.4 节 vs 2.4 节的定位**

| 章节 | 定位 | 内容 | 篇幅 |
|------|------|------|------|
| **1.4 核心计算步骤** | 概览 + 关键要点 | 计算步骤、分支差异对比 | 简化（20-30 行） |
| **2.4 类别特有设计** | 完整实现 | 每个分支的完整伪代码 | 详细（50-100 行/分支） |

**2.4 节编写要求**：
- ✅ **必须写**：每个分支的 Compute 核心计算伪代码（CopyIn/CopyOut 非必须）
- ✅ **必须包含**：Buffer 使用、API 选择、参数含义的详细标注

### 7.2 禁止使用引用

**⚠️ 重要**：在 2.4 节的分支设计中，**禁止使用引用**代替伪代码：

```
❌ 错误：**核心流程伪代码**：见 [1.4.1 节](#ref)
✅ 正确：**核心流程伪代码**：[完整伪代码 50+ 行]
```

**原因**：
- 实现者需要在 2.4 节看到完整设计
- 避免"跳来跳去"查阅文档

### 7.3 生成前的自检清单（AI 必读）

**结构完整性**：
- [ ] 是否包含所有必需章节
- [ ] 是否识别算子类别并查阅对应方法论
- [ ] 是否应用成熟方案的关键设计

**分支伪代码完整性（复杂算子强制）**：
- [ ] 1.4 节是否包含核心计算步骤概览
- [ ] **2.4 节每个分支是否直接包含完整核心伪代码（不是引用）**
- [ ] **每个分支的核心伪代码只包含 Compute 核心计算（CopyIn/CopyOut 非必需）**
- [ ] 伪代码是否有 Buffer 使用标注
- [ ] 伪代码是否有 API 选择说明
